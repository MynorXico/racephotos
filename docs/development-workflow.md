# Development Workflow ‚Äî RaceShots

This document describes the end-to-end process for building each user story
in the v1 backlog. The goal is maximum automation: agents handle design, build,
review, and testing while you make the decisions that require human judgment
(merging, resolving open architectural questions, confirming UX direction).

---

## Agents vs commands ‚Äî the distinction

|                       | Commands (`.claude/commands/`)        | Agents (`.claude/agents/`)                    |
| --------------------- | ------------------------------------- | --------------------------------------------- |
| Runs in               | Main conversation                     | Isolated subprocess                           |
| Parallel execution    | No                                    | Yes                                           |
| Context window impact | Consumes main context                 | Own isolated context                          |
| Tool access           | All tools                             | Restricted per role (read-only for reviewers) |
| Best for              | Interactive, multi-step orchestration | Specialist, automated, parallelisable         |

**Commands** are the orchestrators and interactive tools (`/ship-feature`,
`/review-story`, `/new-adr`). **Agents** are the specialist reviewers that fire
automatically and run in parallel after a PR is created.

## Per-story flow

```
1. /review-story RS-00X                  ‚Üê tech lead gate (interactive)
2. /ship-feature RS-00X                  ‚Üê full build loop (automated)
     ‚îú‚îÄ‚îÄ ux-designer agent               ‚Üê UI stories only, runs before build
     ‚îú‚îÄ‚îÄ build-feature                   ‚Üê interface ‚Üí tests ‚Üí impl ‚Üí CDK
     ‚îú‚îÄ‚îÄ make test-unit / lint / synth   ‚Üê validation
     ‚îú‚îÄ‚îÄ open PR
     ‚îî‚îÄ‚îÄ [in parallel]
           ‚îú‚îÄ‚îÄ security-reviewer agent
           ‚îú‚îÄ‚îÄ db-expert agent
           ‚îú‚îÄ‚îÄ perf-reviewer agent
           ‚îî‚îÄ‚îÄ qa-expert agent
3. Fix any CHANGES REQUIRED from agents  ‚Üê iterate until all APPROVED
4. Merge ‚Üí next story
```

### Step 1 ‚Äî `/review-story RS-00X`

Run this before starting any build. Catches issues before code is written:

- Template compliance (all sections present, ACs in Given/When/Then)
- Infrastructure naming (no `{envName}` on account-scoped resources)
- REST correctness (no GET side effects, idempotency documented)
- ADR references exist and are resolved
- DoD checklist intact

**Verdict: READY TO BUILD or NEEDS CHANGES.** Fix NEEDS CHANGES first.

### Step 2 ‚Äî `/ship-feature RS-00X`

Runs the full automated loop (9 steps):

1. Locate story, confirm `Status: ready`
2. Check ADR dependencies
3. Run `ux-designer` agent if `Has UI: yes` and no spec exists yet
4. Build: interface ‚Üí tests ‚Üí implementation ‚Üí CDK construct
5. Validate: `make test-unit`, `make lint`, `make synth` (+ Angular checks for UI)
6. Open PR
7. Run `security-reviewer`, `db-expert`, `perf-reviewer`, `qa-expert` **in parallel**; post results as PR comments; fix any `CHANGES REQUIRED`
8. Execute test plan, post results as PR comment
9. Set `Status: done`

**If the build hits an ambiguous design decision:** agent stops, runs
`/new-adr "<question>"`, waits for you to confirm the ADR, then resumes.
Never makes a silent architectural assumption.

### Step 3 ‚Äî Fix and merge

Review the PR and agent comments. Merge once all agents show `APPROVED`.

### Step 4 ‚Äî Monitor the CodePipeline execution

Every merge to `main` triggers `racephotos-pipeline` in the Tools account. Always follow up:

#### 4a ‚Äî Find the triggered execution

```bash
aws codepipeline list-pipeline-executions \
  --pipeline-name racephotos-pipeline \
  --max-items 5 \
  --profile tools_readonly \
  --query 'pipelineExecutionSummaries[*].{id:pipelineExecutionId,status:status,startTime:startTime}' \
  --output json
```

Match the execution whose `startTime` is closest to the merge time.

#### 4b ‚Äî Poll until a terminal state

```bash
aws codepipeline list-action-executions \
  --pipeline-name racephotos-pipeline \
  --filter pipelineExecutionId=<ID> \
  --profile tools_readonly \
  --query 'actionExecutionDetails[*].{stage:stageName,action:actionName,status:status}' \
  --output json
```

Poll every ~30 s. Terminal states: `Succeeded`, `Failed`, `Cancelled`.

#### 4c ‚Äî Interpret `Cancelled`

If overall = `Cancelled` **and** both Synth and SelfMutate = `Succeeded` ‚Üí this is normal self-mutation. The pipeline rewrote itself and restarted. Find the **next** execution in `list-pipeline-executions` and monitor that one instead.

#### 4d ‚Äî If `Failed`

1. Note the failed stage/action from `list-action-executions`.
2. Fetch the CodeBuild logs:
   ```bash
   aws logs get-log-events \
     --log-group-name /aws/codebuild/<project-name> \
     --log-stream-name <stream-id> \
     --profile tools_readonly
   ```
3. Diagnose the root cause from the log output.
4. Create a dedicated fix branch **from `main`** (never reuse a feature branch):
   ```bash
   git checkout main && git pull
   git checkout -b fix/<story>-<short-description>
   ```
5. Apply the fix, validate locally (`make cdk-check` / `npm test` / `go test`), commit, push, and open a PR:
   ```bash
   gh pr create --title "fix(...): ..." --body "..."
   ```
6. **Never merge the fix PR automatically.** Present the PR URL and wait for explicit human approval before merging.
7. After approval ‚Üí merge ‚Üí return to step 4a.

## Adding a new specialist agent

1. Create `.claude/agents/<name>.md` with this frontmatter:

   ```markdown
   ---
   name: agent-name
   description: When to use this agent (shown in agent picker and used by ship-feature to decide when to invoke)
   tools: Read, Grep, Glob   ‚Üê restrict to minimum needed; no Write unless agent produces output files
   ---

   System prompt ‚Äî role, what to read, checks to run, output format with APPROVED | CHANGES REQUIRED verdict
   ```

2. Add a row to the agent table in this document
3. Add the invocation condition to Step 7 of `.claude/commands/ship-feature.md`

## Handling mid-build architectural decisions

When `/ship-feature` or any agent encounters a design question not covered by
the story or an existing ADR:

```
Stop.
Run: /new-adr "<the open question>"
The skill will:
  - Load PRODUCT_CONTEXT.md and all existing ADRs
  - Enumerate options with pros/cons
  - Recommend a decision
  - Write the ADR to docs/adr/
  - Update affected stories
Resume build once the ADR is accepted.
```

## Bug flow

Bugs follow the same branch ‚Üí PR ‚Üí pipeline path as stories. The difference is
they start with a failing test rather than a story document.

```
1. GitHub Issue opened (reporter ‚Äî public entry point)
2. Maintainer triages: assigns severity, creates docs/bugs/BUG-NNN-<slug>.md
3. Fix branch created from main: fix/<slug>
4. Failing unit test written that reproduces the bug
5. Fix implemented ‚Äî test passes
6. make validate (full suite)
7. PR opened ‚Äî references bug doc + GitHub Issue (Fixes #<issue>)
8. CI must be green; same specialist agents fire as for stories
9. Merge (explicit human approval only) ‚Üí monitor pipeline
10. GitHub Issue closed
```

### Severity levels

| Level       | Meaning                                                        | Target response |
| ----------- | -------------------------------------------------------------- | --------------- |
| P1-critical | Data loss, broken core path (upload/search/download/purchase)  | Same day        |
| P2-high     | Feature broken but workaround exists                           | Next PR         |
| P3-low      | Polish issue, minor UX gap                                     | Backlog         |

### Bug doc

Every bug picked up by a maintainer gets a `docs/bugs/BUG-NNN-<slug>.md` created
from `docs/bugs/TEMPLATE.md`. This doc is the internal counterpart to the GitHub
Issue ‚Äî it holds the root cause analysis, acceptance criteria, and DoD checklist.

```bash
cp docs/bugs/TEMPLATE.md docs/bugs/BUG-001-<slug>.md
# fill in: GitHub Issue number, severity, affected story, root cause, ACs
```

### Fix branch naming

```bash
git checkout main && git pull
git checkout -b fix/<slug>
# example: fix/search-empty-bib-returns-500
```

## Key constraints (enforced by `/review-story` and `security-reviewer`)

- No `{envName}` suffix on account-scoped resources (DynamoDB, SQS, Cognito, SES templates)
- S3 bucket names always include `{envName}` (globally unique)
- One Lambda per HTTP method ‚Äî never a router Lambda
- All env vars use `RACEPHOTOS_` prefix; `os.Getenv` only in `main.go`
- Every SQS-triggered Lambda has a DLQ + CloudWatch alarm
- No raw AWS SDK errors returned to API Gateway callers
- No runner PII (email, bank details, payment references, presigned URLs) in logs
- `context.Background()` never inside a handler or anything it calls
- All DynamoDB expressions use `expression.Builder` ‚Äî no string concatenation
