# UX Spec — RS-012: Runner downloads a photo via download token

**Story**: RS-012  
**Routes**: `/download/:token` (public), `/redownload` (public)  
**Persona**: Runner — casual user, mobile-first, goal-oriented (they followed a link from email or are trying to recover a lost one)  
**Written**: 2026-04-18

---

## Overview

Two lightweight, stateless components handle the post-purchase download experience.
Neither component uses NgRx — there is no shared state, no optimistic update, and
no data that survives navigation. Each component manages its own local state using
Angular signals or component-level properties.

`DownloadRedirectComponent` is the landing page for download links. Its only job
is to redirect the browser to the presigned S3 URL generated by the Lambda. The
runner sees a brief spinner while the redirect is in-flight; if the API returns
404 they see an error message with a link to `/redownload`.

`RedownloadRequestComponent` is a simple recovery form. The runner enters their
email and the system re-sends all their active download links. The component shows
a generic success message on 200 and a rate-limit message on 429.

Both routes are public — no `canActivate` guard and no `AuthInterceptor` header
injection (the `AuthInterceptor` already only attaches headers when a session
exists; it will pass through silently for unauthenticated callers).

## DownloadRedirectComponent (`src/app/events/download/download-redirect.component.ts`)

### Purpose

Navigates the browser to the API's presigned S3 download URL for the token in the
route parameter, showing a loading state while the redirect is pending and an
error state if the token is invalid or the purchase is not approved.

### Layout

The component occupies the full viewport. All content is vertically and
horizontally centred on screen (CSS flexbox column, `justify-content: center`,
`align-items: center`). There is no page header or navigation — this page exists
solely to initiate a file download and then be left behind.

**Loading state** (default on arrival):
- A `MatProgressSpinner` (mode `indeterminate`, diameter 48) centred on screen
- Below the spinner: the text "Preparing your download…" in `mat-body-1` typography,
  secondary text colour
- No other controls are visible

**Error state** (API returns 404 or network failure):
- A `MatIcon` of `link_off` at 48px in the M3 `error` colour token, centred
- Below the icon: the heading "Download link not found" in `mat-title` typography
- Below the heading: the message "This download link is invalid or has already
  been used. If you believe this is an error, you can request a new link." in
  `mat-body-1` typography, secondary colour, centred, max-width 320px
- Below the message: a `MatButton` (raised, primary) labelled "Request a new link"
  that navigates to `/redownload` via Angular `Router.navigate`

There is no success/populated state: on success the browser is immediately
redirected away from this page via `window.location.href` assignment and the
component never renders a success view.

### States

**Loading** (initial, covers the duration of the API call):
- Full-viewport centred layout
- `MatProgressSpinner` indeterminate, diameter 48
- Label "Preparing your download…" below the spinner
- No buttons or links visible

**Error — 404 or network failure**:
- Spinner replaced by `link_off` icon (48px, error colour) + heading + body text +
  "Request a new link" button
- The `<main>` container has `role="alert"` so screen readers announce the error
  without the runner needing to navigate to it

**Success**: component never renders a visible success state. The redirect is
triggered by assigning `window.location.href` in `ngOnInit` (or an effect) after
the component resolves the token from the route. The browser leaves the page
immediately.

**N/A states**: Empty state is not applicable — the route always has a token
parameter and the loading/error dichotomy fully covers the outcome space.

### Redirect mechanics

On `ngOnInit`, the component reads the `:token` route parameter from
`ActivatedRoute.snapshot.paramMap.get('token')` and issues a `GET` request to
`${apiBaseUrl}/download/${token}` using Angular `HttpClient`.

The API Lambda responds with `200 { "url": "<presignedS3Url>" }` on success and
404 on failure. The component:

1. Shows the spinner immediately on mount.
2. Calls `HttpClient.get<{ url: string }>(\`${apiBaseUrl}/download/${token}\`)`.
   - On 200: assign `window.location.href = response.url` to trigger the browser download.
   - On 404: set local `state = 'error'`.
   - On any other HTTP error or network failure: set local `state = 'error'`.
3. On network error: set local `state = 'error'`.

The `apiBaseUrl` is read from the runtime `config.json` using the existing
`ConfigService` (or equivalent) already in use by other components that call the
API.

### Responsive behaviour

**375px (mobile)**:
- The centred block uses `padding: 24px` so content does not touch screen edges
- The error message paragraph is full-width up to its `max-width: 320px` cap
- "Request a new link" button is full-width (`width: 100%`), minimum height 48px
  to meet touch target requirements
- Spinner diameter remains 48px

**1280px (desktop)**:
- Layout is identical — full-viewport centred — because this page has no content
  beyond the single focal element
- "Request a new link" button width follows Material defaults (auto-width, fitted
  to label) rather than full-width, since pointer precision is available

### Accessibility

- The page `<title>` is "Downloading your photo — RaceShots" while loading;
  updates to "Download error — RaceShots" in the error state
- The spinner container has `aria-label="Preparing your download"` and
  `aria-live="polite"` so screen readers announce the loading state
- The error container has `role="alert"` so the error is announced immediately
  when it replaces the spinner, without requiring focus movement
- The error heading ("Download link not found") is an `<h1>` — it is the only
  heading on the page
- The "Request a new link" button has sufficient colour contrast and a visible
  focus ring (Material default)
- The `link_off` icon is decorative alongside the heading — it has `aria-hidden="true"`

### Angular Material components to use

- `MatProgressSpinnerModule` (`mat-spinner`, mode `indeterminate`, diameter 48)
- `MatIconModule` (`link_off`, `aria-hidden="true"`) for the error state
- `MatButtonModule` (raised, primary) for "Request a new link"

### NgRx integration

This component uses no NgRx. State is managed with three component-level values:

```typescript
state: 'loading' | 'error' = 'loading';
```

No selectors, no actions, no store injection. The `ConfigService` (or
`APP_CONFIG` injection token, whichever pattern the existing codebase uses) is
injected to read `apiBaseUrl` at runtime.

## RedownloadRequestComponent (`src/app/events/download/redownload-request.component.ts`)

### Purpose

Provides a recovery form where a runner enters their email address to receive a
fresh copy of all their active download links. Handles success (200), rate-limit
(429), and network error outcomes.

### Layout

The component renders a single centred card. At mobile the card spans the full
viewport width with padding; at desktop it is constrained to a maximum of 480px
and centred with auto horizontal margins.

**Inside the `MatCard`**, top to bottom:

1. **Card header** (`mat-card-header`):
   - Title: "Resend download links" (`mat-card-title`)
   - Subtitle: "Enter the email address you used when purchasing your photos."
     (`mat-card-subtitle`)

2. **Card content** (`mat-card-content`):
   - A single `MatFormField` (appearance `outline`) containing an `<input
     matInput type="email">` labelled "Email address"
   - Hint text below the field (via `<mat-hint>`): "We'll send links to all
     approved purchases for this address."
   - Validation: `Validators.required` and `Validators.email`
   - Required error message: "Email is required." linked via `aria-describedby`
   - Invalid format error message: "Enter a valid email address." linked via
     `aria-describedby`
   - The form field is disabled while the request is in-flight

3. **Card actions** (`mat-card-actions`, `align="end"`):
   - A single `MatButton` (raised, primary) labelled "Send links"
   - While the request is in-flight: the button is disabled and shows a
     `MatProgressSpinner` (mode `indeterminate`, diameter 20, inline) in place of
     the button label. The button width does not collapse when switching to
     spinner — use a fixed minimum width or a flex layout that maintains the
     button's resting size. The button text "Send links" is hidden with
     `visibility: hidden` (not `display: none`) during loading to preserve layout.

4. **Outcome message area** (below card actions, inside the card):
   Conditionally rendered; only one of the three variants appears at a time.

   - **Success (200)**: a `MatIcon` of `check_circle` in the M3 `primary` colour
     token (16px inline), followed by the text "If we have purchases for that
     email, you'll receive a link shortly." in `mat-body-2` typography. The
     container has `role="status"` so screen readers announce it on appearance.

   - **Rate limit (429)**: a `MatIcon` of `timer` in the M3 `tertiary` or
     `secondary` colour token (16px inline), followed by the text "Too many
     attempts. Please wait an hour and try again." in `mat-body-2` typography.
     The container has `role="alert"`.

   - **Network error (other non-429 failure)**: a `MatIcon` of `error_outline` in
     the M3 `error` colour token (16px inline), followed by the text "Something
     went wrong. Please try again." in `mat-body-2` typography. The container has
     `role="alert"`. After a network error the form and button remain enabled so
     the runner can retry immediately without reloading the page.

### States

**Loading** (request in-flight):
- Form field disabled
- "Send links" button disabled, spinner shown inline in button, button label hidden
- No outcome message shown (any previous outcome message is cleared on new
  submission)

**Empty / default** (page first loads, no submission yet):
- Form field enabled, empty
- "Send links" button enabled with label text
- No outcome message

**Success** (API returns 200):
- Form field re-enabled (runner can submit again for a different email if needed)
- "Send links" button re-enabled
- Success outcome message shown

**Rate limit (429)**:
- Form field re-enabled
- "Send links" button re-enabled
- Rate-limit outcome message shown

**Error (non-429 failure)**:
- Form field re-enabled
- "Send links" button re-enabled
- Network error outcome message shown

**Error — form validation** (runner submits with invalid/empty email):
- `markAllAsTouched()` is called before the API request so Material shows the
  inline validation errors on the form field immediately
- No API call is made while the form is invalid
- No outcome message is shown for a validation failure; the inline field errors
  are sufficient

### Responsive behaviour

**375px (mobile)**:
- The `MatCard` spans the full viewport width (`width: 100%`) with no border
  radius (or a subtle 4px radius only on top edges if a bottom-anchored feel is
  desired — see UX-D3 below) and 16px horizontal padding inside the card content
- The "Send links" button is full-width (`width: 100%`), minimum height 48px
- Page-level padding: 0px — card fills edge-to-edge on small screens
- Card title uses `mat-headline-6` typography; card subtitle uses `mat-body-2`

**1280px (desktop)**:
- The card is centred with `max-width: 480px` and `margin: 64px auto`
- Card uses the default `MatCard` border radius and elevation
- The "Send links" button follows Material defaults (auto-width, aligned to the
  right via `align="end"`)
- The outcome message area sits below the actions row, left-aligned within the
  card content

### Accessibility

- The page `<title>` is "Resend download links — RaceShots"
- The card title "Resend download links" is rendered as an `<h1>` inside the
  `mat-card-title` — it is the sole heading on the page
- The email input has a visible label "Email address" and does not rely on
  placeholder alone
- Both validation error messages (`mat-error`) are linked to the input via
  `aria-describedby` (Angular Material handles this automatically when
  `mat-error` is used inside `mat-form-field`)
- The submit button has `aria-busy="true"` while the request is in-flight
- Outcome messages use `role="status"` (success, neutral) or `role="alert"`
  (error, rate-limit) to announce themselves to screen readers without
  requiring focus movement
- Colour is never the sole indicator of outcome state — each message uses an
  icon (`check_circle`, `timer`, `error_outline`) alongside colour

### Angular Material components to use

- `MatCardModule` (`mat-card`, `mat-card-header`, `mat-card-title`,
  `mat-card-subtitle`, `mat-card-content`, `mat-card-actions`)
- `MatFormFieldModule` (appearance `outline`) for the email field
- `MatInputModule` for the `<input matInput type="email">`
- `MatButtonModule` (raised, primary) for "Send links"
- `MatProgressSpinnerModule` (mode `indeterminate`, diameter 20) inline in the
  button
- `MatIconModule` (`check_circle`, `timer`, `error_outline`) for outcome messages
- `ReactiveFormsModule` — `FormGroup` / `FormControl` with `Validators.required`
  and `Validators.email`

### NgRx integration

This component uses no NgRx. State is managed with component-level properties:

```typescript
submitState: 'idle' | 'loading' | 'success' | 'rate-limited' | 'error' = 'idle';
```

The component injects `HttpClient` (or a thin `RedownloadService` wrapper that
the build agent creates alongside the component) and calls
`POST ${apiBaseUrl}/purchases/redownload-resend` with body `{ email: string }`.

- 200 response → set `submitState = 'success'`
- 429 response → set `submitState = 'rate-limited'`
- Any other HTTP error or network failure → set `submitState = 'error'`

The `apiBaseUrl` is read from the same `ConfigService` / `APP_CONFIG` injection
token used by `DownloadRedirectComponent`.

## Route registration

Add to `app.routes.ts` — no `canActivate` guard, no NgRx providers:

```typescript
{
  path: 'download/:token',
  loadComponent: () =>
    import('./events/download/download-redirect.component').then(
      (m) => m.DownloadRedirectComponent,
    ),
},
{
  path: 'redownload',
  loadComponent: () =>
    import('./events/download/redownload-request.component').then(
      (m) => m.RedownloadRequestComponent,
    ),
},
```

Both routes must be placed above the catch-all `{ path: '', redirectTo: '/login' }`
entry to ensure they match before the fallback fires.

## Storybook stories

### `download-redirect.component.stories.ts`

Three stories under title `Runner/Download/DownloadRedirectComponent`:

| Story name | `state` input | Description |
|---|---|---|
| `Loading` | `'loading'` | Default on arrival — spinner and label |
| `Error` | `'error'` | Invalid/expired token — icon, heading, message, CTA button |

The component does not have a `Success` story because the success path triggers a
`window.location.href` redirect and immediately leaves the page. Storybook cannot
usefully render that. Document this in a `// Note:` comment at the top of the
stories file: "Success state navigates away; no Storybook story is possible."

To make the component Storybook-testable, the build agent must accept `state` as
an `@Input()` that overrides the live API call. In production the component
ignores this input and always starts in `'loading'`, then transitions based on
the API response.

### `redownload-request.component.stories.ts`

Four stories under title `Runner/Download/RedownloadRequestComponent`:

| Story name | `submitState` input | Description |
|---|---|---|
| `Default` | `'idle'` | Empty form, no message |
| `Loading` | `'loading'` | Spinner in button, form disabled |
| `Success` | `'success'` | Success message visible below card actions |
| `RateLimited` | `'rate-limited'` | Rate-limit message visible |
| `NetworkError` | `'error'` | Network error message visible |

To make the component Storybook-testable, `submitState` is accepted as an
`@Input()` that sets the initial rendered state. In production the component
manages `submitState` internally and ignores the input after construction.

## UX decisions not in the story ACs

The following decisions were made by the UX spec author and are not explicitly
stated in the story. Flag these for team review before build begins.

**UX-D1 — Redirect via `window.location.href`, not `HttpClient` follow-through**  
AC5 explicitly states `window.location.href = "${apiBaseUrl}/download/${token}"`.
However, this means the component cannot natively detect a 404 before the redirect
attempt. This spec proposes issuing an `HttpClient.get()` call first (with
`{ observe: 'response' }`) so the component can intercept a 404 before assigning
`window.location.href`. This adds one extra round-trip on the happy path. An
alternative is to perform the `href` assignment unconditionally and rely on the
server-rendered 404 page — but that breaks the Angular error state in AC6. The
team should confirm whether the extra round-trip is acceptable or whether the
Lambda should return JSON on 404 so `HttpClient` can detect it.

**UX-D2 — Error state reuse for network failure vs 404 on DownloadRedirectComponent**  
AC6 only specifies the 404 error message. This spec uses the same error view for
any non-200 API response (5xx, network timeout). The message "This download link
is invalid or has already been used" is accurate for 404 but misleading for a 5xx.
Options: (a) use a generic "Something went wrong, please try again" for 5xx, with
the 404 message reserved for confirmed-404; (b) treat all failures identically as
a link problem. Option (a) is slightly better UX but adds a second error variant.
Confirm with the PO.

**UX-D3 — RedownloadRequestComponent card appearance on mobile**  
The story does not specify whether the card has rounded corners at 375px or
extends edge-to-edge. This spec opts for edge-to-edge (no card border radius,
no card shadow, no horizontal margin) on mobile, matching the style used by the
purchase stepper steps on the same persona's journey. Confirm if the full-bleed
mobile card is the preferred pattern or if a padded floating card is wanted.

**UX-D4 — Storybook testability via `@Input()` override for internal state**  
Neither component has props in the AC. This spec adds a `state` / `submitState`
`@Input()` accepted only for Storybook. This is a common Angular Storybook pattern
but it does slightly complicate the component API. An alternative is to use a
service stub in Storybook providers and never expose internal state as an input.
Confirm the preferred approach with the build agent convention already in use
(see `EmailStepComponent` which uses `@Input() loading` and `@Input() error`
rather than an internal state enum — the same pattern should be followed here for
consistency).

**UX-D5 — "Request a new link" button navigates to `/redownload` via Router**  
AC6 states the error message includes "request a new link at /redownload" but does
not specify whether this is an in-app `routerLink` or a plain `<a href>`. This
spec uses Angular `Router.navigate(['/redownload'])` on button click (not an
anchor tag) to keep the navigation within the SPA. The button is a `MatButton`
rather than a link element, which means it does not render as an `<a>` and will
not be right-click-openable in a new tab. If the team prefers a native link, use
`MatAnchor` (`mat-raised-button` on an `<a>` element with `routerLink="/redownload"`).

**UX-D6 — No explicit "try again" on RedownloadRequestComponent network error**  
AC7 does not specify a retry CTA for network errors. This spec keeps the form
and button enabled after a network error, which allows the runner to retry by
clicking "Send links" again. No separate "Try again" button is added. If a
prominent retry affordance is wanted, consider reusing the `MatButton` (stroked)
pattern from `EventSearchComponent`'s error state.
